## 構成

### 1. LLMと一緒にアウトプットを作るということ

- 1.1 アドベントカレンダーで6本書いた話
  - ポイント: 1ヶ月で6本は大変。Geminiと壁打ちしながら乗り切った
  - 具体例: LLMが深堀りの質問を投げてくれる → 自分の考えで答えていく中で構成が深まる体験。普段は友だちとやっていたが、LLMはいつでもいくらでも質問できる
  - 独自性: 「友だちとの壁打ち → LLMとの壁打ち」という変化の体験

- 1.2 失敗と発見 — 書かせ方で全然違う
  - ポイント: 構成を渡してすぐ書かせたら、LLMが勝手に推測して自由なことを書き始めた。修正が大変だった
  - 具体例: 章・項の整理段階で深堀り質問をもらい、構成をブラッシュアップ → その上で書かせると主張の上書きが少ない。LLMの出した文章は全部読み、違和感があれば細かく修正する
  - 独自性: 「書かせて直す」ではなく「構成を一緒に作ってから書かせる」という試行錯誤から得た知見

- 1.3 LLMに書かせることへの自分の考え
  - ポイント: アウトプットの価値は自分の体験・主張の共有にある。主導権を手放さなければ、LLMは強力なパートナーになる
  - 根拠: 1.2の体験から導いた結論。詳しくはQiita記事「LLMに『書かせる』な、LLMと『書く』んだ」にも書いた（補足リンク: https://qiita.com/yuni0928/items/aad44bcc51050daa11c7 ）
  - 独自性: 失敗体験→成功パターン発見→主張という流れが自分の経験に根ざしている

### 2. だから、output-flow を作った

- 2.1 ワークフロー化のきっかけ
  - ポイント: エンジニア集会の活動や技術の勉強をしつつ、LT会への参加やセッション登壇の準備をする必要に迫られた。アウトプット作りで勉強や活動の時間が削れてしまうのが嫌だった
  - 具体例: 壁打ちの「構成を整理→深堀り→執筆」という成功パターンを、毎回手作業でやるのではなく再利用可能な形に自動化したいと思った
  - 独自性: 「アウトプットしたいけど時間がない」は読者と同じ悩み。そこから自動化に至った実体験

- 2.2 output-flow の紹介
  - ポイント: こうした経緯で、自分のアウトプットを効率化するためのLLMワークフロー「output-flow」を作った（GitHub: https://github.com/yuni-shinogami/output-flow ）
  - 具体例: 1章で見つけた「構成段階で深堀り→ブラッシュアップ→執筆」というパターンをClaude Codeのカスタムスラッシュコマンドとして実装。outlineスキルと記事執筆スキルの2つで構成
  - 補足: このワークフローは作業配信でゼロから作り上げた（配信リンク: https://www.youtube.com/live/VgpXpXksEis ）。実際に作っていく過程も見られる

### 3. output-flow を使ってみよう

- 3.1 アウトラインを作る（/outline）: Phase 1〜4の流れと、それぞれで何をするか
- 3.2 記事を執筆する（/write）: アウトラインから本文を生成し、推敲する流れ

### 4. 自分だけのアウトプット環境を作ろう

- 4.1 スキルの仕組みを覗く
  - ポイント: スキル定義の構造を簡単に紹介。自分だけのツールを作るヒントに

- 4.2 output-flow のこれから
  - ポイント: スライド生成・同人誌原稿など、幅広いアウトプット支援に育てたい（簡潔に）

- 4.3 読者へのメッセージ
  - ポイント: output-flowを作って一番変わったのは、普段やっていることを積極的に自動化しようという姿勢。自動化すると行動のハードルが下がり、無理なくできるようになる。あなたもoutput-flowでアウトプットを始めてみてほしい。感想はXで共有してもらえるととても嬉しい
  - 独自性: ツールの宣伝ではなく、自分が得た変化を読者に手渡す形
