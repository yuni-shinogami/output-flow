# レビュー用サブエージェントプロンプト

以下のテンプレートの `{goal}` `{slide_plan}` `{slides}` に実際の内容を埋め込んで使う。

## 1. ゴールとの整合性チェック

```
以下のゴール定義とスライドを読み、「ゴールとの整合性」を点検してください。

【観点】
- ゴールで定めたメッセージが聴衆に伝わるスライド構成か
- 受け手に期待する行動/変化につながる流れになっているか
- 目的から逸れたスライド、不足しているスライドはないか
- 核メッセージが明確に打ち出されているか

【ゴール定義】
{goal}

【スライド】
{slides}

問題点は具体的なスライド番号を引用して指摘し、改善案を出す。問題なければ「問題なし」と報告。
```

## 2. 視覚的明瞭さチェック

```
以下のスライド設計プランとスライドを読み、「視覚的明瞭さ」を点検してください。

【観点】
- 1スライド1メッセージの原則が守られているか
- テキスト量が多すぎるスライドはないか（箇条書き5項目以内が目安）
- パターンごとの容量制限を超えていないか（.claude/skills/slide/themes/starter/docs/patterns.md の容量の目安を参照）
  - emphasis-detail: h1が2行折り返す場合、箇条書きは2項目が上限
  - content + img-*: 箇条書きは3項目が上限
- 情報密度は適切か（詰め込みすぎ / スカスカ）
- 箇条書きの各項目は1行に収まる長さか（25〜35文字目安。2行に渡る場合は短縮または分割を提案）
- h1が2行に渡る場合、意味のまとまりで改行位置が調整されているか（助詞の途中で切れていないか）
- 孤立文字チェック: h1テキストがパターンの1行文字数上限をわずかに超えて1〜2文字だけ次行にはみ出していないか。上限目安: title=9文字, cover=14文字, emphasis=11文字, bridge=13文字, emphasis-detail=15文字, section-box/section-line=13文字, summary ol li=18文字。超える場合は `<br>` での改行制御または文言短縮を提案する
- スライド設計プランのビジュアル方針と一致しているか
- 聴衆が一目で理解できる構成か
- SVG図解がある場合: サイズ・配置は適切か、画像レイアウトクラスが使われているか

【スライド設計プラン】
{slide_plan}

【スライド】
{slides}

問題点は具体的なスライド番号を引用して指摘し、改善案を出す。問題なければ「問題なし」と報告。
```

## 3. 独自価値の表現チェック（最重要）

```
以下のスライドを読み、「独自価値の表現」を点検してください。これは最も重要なチェックです。

【観点】
- 自分の経験・体験に基づくスライドが十分にあるか
- 一般論や誰でも作れるスライドに埋もれていないか
- 「この人だから話せる」価値が出ているか
- 経験を語るスライドの臨場感・具体性は十分か
- エピソードがスライド向けに適切に要約されているか（長すぎず、薄すぎず）

【スライド】
{slides}

各スライドについて、独自の経験に基づく部分と一般的な記述を分類する。
一般的な記述が目立つスライドには、どう改善すれば独自価値が出るか具体的に提案する。
```

## 4. プレゼンの流れチェック

```
以下のスライドを読み、「プレゼンの流れ」を点検してください。

【観点A: テンポと構成】
- 冒頭で聴衆の関心を引けるか
- 聴衆の集中力が切れそうな箇所はないか（長い説明、単調な展開）
- スライド間のつなぎは自然か（話の橋渡し）
- テンポは適切か（緩急があるか）
- 時間配分は妥当か（スピーカーノートの量から推定）
- 結びは余韻を残せるか

【観点B: スライド構造と演出技法】
- slide-plan.mdの構成パターン（問題解決型/体験談型/PREP型）に合った流れになっているか
- ブリッジスライドの配置: セクション転換にブリッジが適切に使われているか、メッセージは1行で簡潔か
- 段階的ビルドアップ: 複雑な概念や図解に段階的な情報開示が使われているか、ビルドアップの枚数は適切か（多すぎ/少なすぎ）
- お品書きの一貫性: お品書き→転換お品書き→まとめの3項目が対応しているか（お品書きを使う構成の場合）
- クロージング: まとめ→クロージングブリッジ→おまけ/宣伝の流れが自然か
- 掴みの効果: 冒頭の掴みスライドが聴衆の興味を引く内容になっているか

【スライド設計プラン】
{slide_plan}

【スライド】
{slides}

問題点は具体的なスライド番号を引用して指摘し、改善案を出す。問題なければ「問題なし」と報告。
```

## 5. デザインシステム準拠チェック

```
以下のスタイルガイド、パターンカタログ、スライドを読み、「デザインシステム準拠」を点検してください。

【観点】
- 全スライドに `<!-- _class: [パターン名] -->` が付与されているか（メタ情報スライドを除く）
- 使用されているパターン名が .claude/skills/slide/themes/starter/docs/patterns.md に定義された13パターンのいずれかか
- カラーパレット制約に違反していないか（NGパターン参照）
- テキスト量がパターンの推奨範囲内か（箇条書き5項目以内等）
- TailwindCSSユーティリティクラスの使い方が適切か（テーマカラー以外のbg-*やpxベースのtext-*を使っていないか）
- frontmatterに `theme: starter-theme` が設定されているか
- TailwindCSS scriptタグが含まれているか
- SVGカラーパレット制約: SVG内の色がテーマのカラーパレット（オレンジ系・グレー・白・黒）に準拠しているか（`#999`, `#666` 等の中間グレーは NG）
- 外部SVGファイル: `images/` ディレクトリに適切なファイル名で配置されているか
- インラインSVG: 20行以下に収まっているか（超える場合は外部ファイル化すべき）

【スタイルガイド】
{style_guide}

【パターンカタログ】
{patterns}

【スライド】
{slides}

問題点は具体的なスライド番号を引用して指摘し、改善案を出す。問題なければ「問題なし」と報告。
```
