# レビュー用サブエージェントプロンプト

以下のテンプレートの `{goal}` と `{outline}` に実際の内容を埋め込んで使う。
`{review_history}` には過去のレビューログ（refine-log.md）の内容を埋め込む。ログが存在しない場合は「なし」とする。
`input-log.md` が存在する場合、「1. 主張との整合性チェック」のプロンプトに
`{input_log}` としてユーザーの元の発言を渡す。存在しない場合は「なし」とする。

## 1. 主張との整合性チェック

```
以下のゴール定義とアウトラインを読み、「主張との整合性」を点検してください。

【観点】
- ゴールで定めた主張とアウトラインの内容がズレていないか
- 各章・項がゴール達成に貢献しているか
- 不要な内容、足りない内容はないか

【語りのスタンス】
{stance}

スタンスが「一緒に考えたい」の場合、以下の観点を追加で点検すること:
- アウトラインの結論が借り物の一般論に聞こえないか
- 「答えを知っている人」のスタンスに無意識にズレていないか
- 結論に違和感やツッコミどころはないか（あればそれ自体が発見のヒント）

【ゴール定義】
{goal}

【アウトライン】
{outline}

【ユーザーの元の発言（input-log）】
{input_log}

【過去のレビュー履歴】
{review_history}

過去のレビュー履歴がある場合：
- 「対応済み」の指摘は再指摘しないこと
- 「未対応」の指摘はアウトラインを確認し、まだ問題が残っているなら「継続」として報告
- 新たに見つかった問題のみ「新規」として報告

input-log が提供されている場合、以下も追加で点検すること:
- 元の発言と成果物の間にニュアンスのズレがないか
- ユーザーの具体的な言い回しが構造化で丸められていないか
- 曖昧な発言をLLMが「補完」したことで意図から離れていないか
- ズレがあれば「ユーザーは『〇〇』と言ったが、成果物では『△△』」の形式で指摘

問題点は具体的に指摘し改善案を出す。問題なければ「問題なし」と報告。
```

## 2. 話の流れチェック

```
以下のアウトラインを読み、「話の流れ」を点検してください。

- 章・項の順序が自然か
- 飛躍や唐突な展開はないか
- つなぎが必要な箇所はないか
- LTなど時間制約のある発表の場合、1章あたりの項が多すぎないか（原則3つ目安。超える場合は章の分割を検討）
- ただし、章末にその章の内容をラップアップする「まとめ項」があるのは良いパターン。まとめ項を問題として指摘しないこと。まとめ項は3項制限に含めても含めなくてもよい

【アウトライン】
{outline}

【過去のレビュー履歴】
{review_history}
```

## 4. 独自価値チェック（最重要）

```
以下のアウトラインを読み、「独自価値」を点検してください。これは最も重要なチェックです。

【観点】
- 自分の経験・体験に基づく内容になっているか
- 「あなたならでは」の価値があるか
- 検索すれば出てくる情報に終始していないか
- 抽象的な主張に具体的な体験の裏付けがあるか（具体→抽象のペアリング）

【具体と抽象のペアリングについて】
抽象的な概念や主張そのものは悪くない。むしろ、具体的な体験から導かれた抽象的な主張は、聴衆が自分の状況に当てはめて持ち帰れる汎用性を生む。
問題になるのは以下の2パターン：
- 抽象的な主張が浮いている: 具体的な体験の裏付けなく概念だけ語っている
- 具体が具体のまま終わっている: 体験談は面白いが、そこから何が言えるのか（抽象化・一般化）が示されていない

【アウトライン】
{outline}

【過去のレビュー履歴】
{review_history}

過去のレビュー履歴がある場合：
- 「対応済み」の指摘は再指摘しないこと
- 「未対応」の指摘はアウトラインを確認し、まだ問題が残っているなら「継続」として報告
- 新たに見つかった問題のみ「新規」として報告

各章・項について以下を点検：
1. 具体的な体験に基づく部分と抽象的な主張の部分を分類
2. 抽象的な主張に具体的な裏付けがペアになっているか確認
3. 裏付けのない抽象的な主張には、どんな体験を添えれば説得力が出るか提案
4. 具体が抽象化されずに終わっている箇所には、そこから導ける主張やメッセージを提案

【追加観点: 声のトーン】
- 結論が「まとめました感」で終わっていないか
- スタンスが「一緒に考えたい」「体験を共有したい」の場合、断定的すぎる表現がないか
- 語り手の等身大の声が聞こえるか
```
